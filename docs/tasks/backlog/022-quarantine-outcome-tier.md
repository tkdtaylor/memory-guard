# Task 022: Quarantine outcome tier (`validate_write` gains a third outcome; contract re-tracer required)

**Project:** memory-guard
**Created:** 2026-07-14
**Status:** ❌ Not started

> **This is the heaviest task in its batch.** Every other write-gate task to date (013/014) changed which writes get flagged while leaving the two-outcome contract (`allow:true` / `allow:false`) untouched. This task changes the outcome shape itself: `validate_write`'s tracer-validated response (task 011 / ADR-008) gains a third state. Because that shape is **tracer-validated**, this task does not get to declare victory on `go test` alone. It must **re-run memory-guard's own contract tracer** (`contract_tracer_test.go`) against the new shape over the live `serve` socket and update `docs/CONTRACT.md` in the same change. Treat the re-tracer step as a first-class deliverable, not a formality.

## Goal

Give `validate_write` a third outcome, **`quarantine`**, for a write that is suspicious but not certain enough to hard-reject: the entry is stored (redacted, like any other write) but into an isolated quarantine namespace, excluded from every normal `validate_read`, flagged, and retrievable only through a new explicit review verb. Today the gate is binary: `allow:true` (stored, readable) or `allow:false` (rejected, fail-closed, never persists). This task adds the missing middle ground without weakening the existing fail-closed poisoning gate and without turning memory-guard into a second policy engine. Memory-guard ships the mechanism (isolated store scope, the outcome, read-exclusion, a review-retrieval path) and a conservative, narrow, additive default trigger; the eventual decision of what routes where is policy-engine's job, not memory-guard's, once that integration exists.

## Context

- **Why this exists (ASI06 framing):** the write-gate's only two outcomes today are "trust it completely" and "reject it completely." A write that trips a genuinely ambiguous signal, plausible as an ordinary retraction ("pretend the above never happened") or as a light context-reset/injection attempt, currently has to be forced into one bucket or the other. Forcing it to `block` repeats the exact fail-open-on-ordinary-English trap that capped Phase B indefinitely ([ADR-010](../../architecture/decisions/010-injection-recall-approach.md), [ADR-011](../../architecture/decisions/011-cap-injection-recall-at-phase-a.md)); forcing it to `allow` throws the signal away. Quarantine is the third bucket that already existed conceptually in the Phase-B postmortem and has never had a home.
- **The zero-regression design constraint:** the existing fail-closed literal-match gate (`RegexDetector.matchesInjection` via `DetectInjection` via `guard.go::ValidateWrite`'s `block` path) and its measured recall/precision floor (F-006, `poisoningRecallFloor`/`poisoningPrecisionFloor` in `fitness_test.go`, task 014 Phase A: recall 0.8125, precision 0.8667) are not touched, not reclassified, not reweighted by this task. The quarantine trigger is a brand-new, additive `Detector`-seam method, orthogonal to `DetectInjection`/`RedactPII`, so every existing corpus assertion in `poisoning_suite_test.go`, `injection_recall_test.go`, and `detector_corpus_test.go` passes unmodified. This was a deliberate design choice after considering, and rejecting, reclassifying the existing base64/URL decode-then-rescan match (task 014 Phase A) from `block` to `quarantine`: that path is a measured, security-audited recall contributor, and softening it would silently shift F-006's baseline and reopen a settled security review. Don't do that. A brand-new, narrow, additive signal is the safer and smaller diff.
- **Policy boundary (load-bearing, must be in the ADR):** in this ecosystem the decision authority, which findings route to `quarantine` vs `block` vs `allow`, belongs to the `policy-engine` block, not memory-guard. Memory-guard is not the place to grow a second, competing decision engine one regex at a time. This task's `DetectBorderline` trigger is an explicit, narrow, conservative placeholder default, enough to make the three-outcome contract meaningfully exercisable end to end, not a claim that memory-guard now owns injection-classification policy. The ADR must say this in as many words, and a future task wires an actual policy-engine decision hook; that wiring is out of scope here.
- **Why a new `Detector` method, not a new flag value on `DetectInjection`:** an interface addition is mechanical but explicit (every implementor must satisfy it, so nothing silently falls through) and keeps the fail-closed `injection_suspected` path's semantics exactly what they've always been. Adding a same-method second flag was considered and rejected: it would require re-deriving `poisoningRecallFloor`'s recall/precision definition mid-task (is a quarantined case a "catch" or a "miss"?) for a floor that a prior, separately-audited task already settled. `DetectBorderline` is a second, independent seam method precisely so that question never has to be re-litigated.
- **In-repo `Detector` implementors (the interface-addition blast radius, confirmed exhaustive):** `RegexDetector` / `NativeDetector` / `PresidioDetector` (`detector.go`, `detector_presidio.go`, real or delegating implementations) plus three test doubles that must gain a trivial `return nil` stub to keep compiling: `slowDetector` / `zeroRecallDetector` (`fitness_test.go`, `//go:build fitness`) and `alwaysAllowDetector` (`poisoning_suite_test.go`). No consumer outside this repo implements `Detector`.
- **Security posture, unchanged:** the read-exclusion and the review path add no new authorization model beyond the existing `0600`-socket-only boundary (the same documented scoping choice `interfaces.md` already notes for `verify_delete`). `review_quarantine(id)` is reachable by anyone who can reach the socket, same as every other verb today. That gap is pre-existing and explicitly out of scope here, not newly introduced.
- Reference: [`docs/CONTRACT.md`](../../CONTRACT.md) (the shape under change), [`docs/tasks/completed/011-contract-tracer-bullet.md`](../completed/011-contract-tracer-bullet.md) (why the re-tracer step is non-negotiable), [`docs/tasks/completed/016-durable-identity-isolation.md`](../completed/016-durable-identity-isolation.md) (the `ScanScoped`/store-scoping pattern this task reuses the shape of, not the identity axis), [`docs/tasks/completed/014-injection-recall-lift-phased.md`](../completed/014-injection-recall-lift-phased.md) plus ADR-010/ADR-011 (why a new pattern needs security-auditor, and why Phase B's rejected patterns are not a shortcut into quarantine; reusing them was considered and rejected, see above), `guard.go` (`ValidateWrite` / `ValidateRead`), `detector.go` (the `Detector` interface), `store_file.go` (the only store adapter needing a wire-format change), `ipc.go` (op dispatch).

## Contract shapes (additive options, ADR must record the choice)

`validate_write`'s tracer-validated response gains a state. Two additive options, both non-breaking at the JSON level (existing consumers reading only known fields are unaffected):

- **Option A, add a boolean:** `{allow, stored_id, flags, quarantined: bool}`. `quarantined:true` implies the write was stored but isolated; `allow` stays `true` for both the allow and quarantine cases. Rejected as the recommended option (the ADR may still choose it, but must record why if so): a client that reads only `allow`/`stored_id` cannot tell "stored and normally readable" from "stored and invisible" without also checking a second, independent boolean that could, in a future 4th-tier world, need to become a second-and-third boolean.
- **Option B, recommended, add a `state` field:** `{allow, stored_id, flags, state: "allow"|"quarantine"|"block"}`, with `allow := (state != "block")` computed for legacy-reader back-compat and `stored_id` non-null for both `allow` and `quarantine` (both states persist an entry; only `block` returns `stored_id:null`). A single source of truth for the outcome, reads as a tri-state, and extends to a future state without another boolean. This mirrors the `label: "poisoning"/"benign"` vocabulary already used in this repo's own test suites.

The ADR must analyze both, record the chosen shape verbatim, and record the rejected alternative's reasoning, the same rigor task 016's ADR-013 gave `ScanScoped`.

A new verb, not previously in the contract, exposes the review-retrieval path:

```
review_quarantine(id) -> { found, content_redacted, flags }
```

`found:false` for an unknown id or for an id that exists but is not quarantined (the review path is quarantine-only; it must never become a generic id-lookup bypass of `validate_read`'s scoping). `content_redacted` is PII-redacted on the way out, same defense-in-depth policy as `validate_read`.

## Out of scope

- **The actual quarantine-vs-block-vs-allow decision policy:** this task ships a narrow, conservative default trigger and the mechanism; the real decision authority is policy-engine's, wired in a future cross-block integration task. Do not grow `DetectBorderline` into a general classifier here.
- **Reusing the rejected Phase-B negation-scope patterns** (ADR-010/ADR-011) as the quarantine trigger: considered and explicitly rejected in Context. Those patterns failed adversarial audit 4 consecutive rounds as a blocking signal and are not a shortcut into a softer tier either without the same audit rigor Phase B never cleared. A future task may revisit this once policy-engine owns the decision, not this one.
- **Reclassifying the existing base64/URL decode-then-rescan match** (task 014 Phase A) from `block` to `quarantine`: considered and rejected, see Context. That path stays a hard block, unchanged.
- **Promotion/demotion of a quarantined entry** back to normal storage or to a hard delete after review: `review_quarantine` is read-only in v0. A follow-up task wires an operator/policy-engine decision verb.
- **Per-op authorization/ACL** on `review_quarantine` beyond the existing `0600`-socket boundary: a pre-existing, documented gap (`interfaces.md`'s `SO_PEERCRED` note), not newly introduced here.
- **A dedicated quarantine OCSF audit-event category** (`audit.go`/`audit_trail_sink.go`): out of scope; the existing best-effort, default-off emission framework is untouched by this task.
- **A bulk/list `review_quarantine` verb:** v0 is single-id lookup only.
- **TTL/expiry of quarantined entries.**
