## Verification ladder

A task earns üü° at levels 1‚Äì4 and ‚úÖ only at level 5 or 6. The `Verified by` column records which level the row reached.

| Level | Evidence | Status this earns |
|-------|----------|-------------------|
| 1 | Code merged | üü° |
| 2 | Unit tests pass (paste verbatim final line of `go test ./...`) | üü° |
| 3 | `make fitness` passes (verbatim closing line) | üü° |
| 4 | CI passes (`gh run watch <id> --exit-status` ‚Üí success) | üü° |
| 5 | **Validation harness** exercises the live runtime path end-to-end ‚Äî paste the command and the final assertion line | ‚úÖ |
| 6 | **Operator-observed** ‚Äî operator (or executor via `go run .` / a live `serve`) saw the targeted behaviour in stdout / logs | ‚úÖ |

If the task targets runtime-observable behaviour (CLI args, server endpoints, file outputs, side effects), level 5 or 6 is **required** before flipping to ‚úÖ. If the task only adds an internal helper covered by unit tests, level 2 may be sufficient ‚Äî but in that case the row's `Verified by` should explicitly say "unit-test-only; no runtime surface" so future readers don't mistake silence for verification.

## Rule

**The task-executor commits at üü° by default.** Only the main session (after spec-verifier APPROVE + the appropriate level-5/6 evidence) updates the row to ‚úÖ, in a separate commit titled `verify: confirm task NNN ‚Äî <level-5/6 evidence>`. This keeps the verification step visible in git history and prevents "merged ‚âÝ done" drift.
