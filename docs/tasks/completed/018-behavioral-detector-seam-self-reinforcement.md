# Task 018: Behavioral-detector seam + `SelfReinforcementDetector`

**Project:** memory-guard
**Created:** 2026-07-14
**Status:** âťŚ Not started

## Goal

Introduce a **second, stateful detection seam**, `WriteInspector`, alongside the existing stateless `Detector` (`detector.go`), and ship its first implementation: a `SelfReinforcementDetector` that flags an agent poisoning itself via repetitive self-authored writes whose pairwise content similarity exceeds a threshold within a cooldown window. Findings surface as an additive value (`self_reinforcement_suspected`) on the existing `validate_write` `flags` array; the tracer-validated contract shape `{allow, stored_id, flags}` stays byte-for-byte unchanged, and this task does not change allow/block behavior for the new flag. The `Detector` interface stays byte-for-byte untouched. Similarity is computed stdlib-only (token-set/Jaccard); `go.mod` stays require-free.

## Context

- **Why `Detector` cannot own this:** `Detector` (`detector.go`) is a pure-function seam over a single text (`RedactPII(text)`, `DetectInjection(text)`). Self-reinforcement is inherently cross-write: it needs the prior writes for an identity, a similarity measure between them, and a cooldown window. Bolting state onto `Detector` would break the "pure function of one text" property every existing `Detector` implementation (`RegexDetector`, `NativeDetector`, `PresidioDetector`) relies on, and would force every backend swap to also carry write-history plumbing it doesn't need. A second seam keeps both concerns swappable independently, following the pattern already established twice in this repo: `MemoryStore` (ADR-005) for storage and `AuditSink` (ADR-007) for emission are each their own seam rather than folded into `Detector`.
- **What "self-reinforcement" means here:** an ASI06-adjacent failure mode where an agent's own prior writes to memory get echoed/paraphrased back into new writes, amplifying an error or an injected belief through repetition rather than through a single injected payload (which the existing `injection_suspected` path already covers). The signal is behavioral (a repetition pattern over time), not lexical (no single write looks like an attack).
- **The seam's shape:** `WriteInspector` sees `(content string, ctx WriteContext)`, where `WriteContext` carries at minimum the writer's normalized identity key (`Principal.Subject()`/`unboundKey`, the same key `principal.go` already derives) and a source-class hint (see the task-020 fallback below). It returns `[]string` flags, mirroring `Detector.DetectInjection`'s shape. Unlike `Detector`, an implementation is explicitly permitted to hold state (a bounded per-identity write history), because that state is the entire point: there is no way to detect self-reinforcement without it. The state lives **inside** the seam implementation, not in `guard.go` or the `MemoryStore` (this is detection-internal working state, not persisted agent memory content).
- **The policy boundary this task respects:** the write-gate's existing invariant is fail-closed on `injection_suspected` (`guard.go::ValidateWrite`; AGENTS.md's write-gate invariant). Whether `self_reinforcement_suspected` should ever become blocking, quarantined, or routed to a human review queue is a **policy-engine decision**, not this task's. This task's contract is: compute the signal honestly, surface it as a flag, and store the write exactly as it would have stored without this task (fail-open on the new flag; fail-closed remains reserved for the existing injection path). A future quarantine-outcome task (referenced in planning as task 022, number to be confirmed at execution time if it has not yet landed) is expected to consume this flag and decide the allow/block policy; this task does not anticipate or implement that policy.
- **The task-020 dependency, and why 018 does not block on it:** durable write-provenance / source-class tagging (referenced in planning as task 020, number to be confirmed at execution time) is expected to supply a trustworthy `agent_authored` vs. `human_authored` signal on every write. That task has not landed. This task defines its own **soft fallback** so it is startable independently (see REQ-007): it reads an optional `identity["source_class"]` string field, the anticipated wire shape task 020 is expected to formalize, and when the field is absent it defaults the write to `agent_authored` (fail-closed toward scrutiny, consistent with this project's fail-closed posture elsewhere). When task 020 lands and starts sending the field on every write, 018's logic needs no change: the fallback branch simply stops firing.
- **Precedent for a fourth seam and an opt-in builder:** `AuditSink` (ADR-007) is wired via a builder method (`(*MemoryGuard).WithAudit`) that returns a modified copy of the guard, defaults to disabled (nil sink), and leaves every pre-existing constructor call site (`NewMemoryGuard(det)`, `NewMemoryGuard(det, store)`) compiling and behaving unchanged. `WriteInspector` follows the same shape (`WithWriteInspector`), so this task cannot regress any existing test that constructs a guard without opting in.
- **A known, deliberately out-of-scope gap:** `guard.go::ValidateWrite` only emits an audit-trail event when `len(piiFlags) > 0` (`if len(piiFlags) > 0 { emitSafe(...) }`). A write flagged `self_reinforcement_suspected` with no PII present therefore emits **no** audit-trail event under today's gating. This task does not change that gate or add a dedicated audit builder for the new flag class; it documents the gap in the ADR (REQ-008) so it is a known, recorded limitation rather than a silent one.

## Out of scope

- Deciding whether `self_reinforcement_suspected` blocks, quarantines, or escalates a write: that is a policy-engine decision, expected to be picked up by a future quarantine-outcome task (referenced in planning as task 022).
- Implementing durable write-provenance / source-class tagging itself: that is task 020's scope. This task consumes an anticipated wire field and defines a documented fallback for its absence.
- Wiring a dedicated audit-trail event class (e.g. a `BuildSelfReinforcementEvent`) or broadening the existing `len(piiFlags) > 0` emission gate so a PII-free self-reinforcement flag reaches audit-trail: documented as a known gap in the ADR, not fixed here.
- Cross-identity or cross-tenant self-reinforcement correlation (e.g. two different agent identities colluding to reinforce each other): this task's history is scoped to a single subject key.
- Semantic/embedding-based similarity (would require a model dependency): the stdlib token-set method is the whole of this task's similarity story; a richer method is a future ADR if the token-set precision proves insufficient in practice.
- Bounding the total number of distinct tracked identities (a memory-exhaustion hardening concern distinct from bounding one identity's history): noted as a follow-up hardening consideration, not required here.
